# Requirements: Daily Derived Stats

## Introduction

Three reading-derived per-day stats â€” the five-block daily-usage breakdown (`findDailyUsage`), the minimum-SOC point of the day (`findMinSOC`), and the top peak-period list (`findPeakPeriods`) â€” are currently recomputed live in the Lambda every time `/day` is called, which is sustainable for single-day requests but does not scale to multi-day rollups because `flux-readings` has a 30-day TTL and a 30-day rollup that re-fetches readings would page through ~258k items per call. This spec moves those three computations into the poller as a once-an-hour pass over yesterday's readings, persists the results on the existing `flux-daily-energy` row, and updates `/day` and `/history` to read from storage for past dates while keeping live computation for today. The feature unblocks T-1022 (history-daily-usage) and any future History rollup that wants per-day SOC lows or peak periods.

## Non-Goals

- Backfilling pre-existing dates whose readings have already aged out of `flux-readings` (pre-feature rows simply lack the new fields and serve as the steady-state for the boundary case).
- Splitting derived stats into a separate `flux-daily-blocks` or `flux-daily-stats` table (the existing `flux-daily-energy` row carries them).
- Adding new iOS UI â€” T-1022 owns that work and is the consuming feature.
- Persisting derived stats for today (today is always live-computed by `/day` and `/history`).
- Changing the schedule of any other poller pass (live, daily-power, system-info, off-peak).
- Adding any new derived stat beyond the three named in this spec.
- Adding a manual "recompute this date" admin endpoint.
- Multi-system support in the poller (the deployment polls one configured system serial; the (sysSn, date) key shape is preserved on disk for future extension but no iteration is added now).
- A schema-version marker on the persisted derived stats â€” algorithm changes that require recomputation will be handled out-of-band when they happen, not pre-engineered here.

## Definitions

- **derivedStats**: the three reading-derived per-day fields produced by `findDailyUsage`, `findMinSOC`, and `findPeakPeriods` against the readings of one calendar date in the poller's configured timezone (`cfg.Location`, currently `Australia/Sydney`). Treated as a single conceptual unit even though they are written as three independent attributes on the row.
- **completed date**: any calendar date strictly before the request's "today" in `Australia/Sydney`.
- **today**: the calendar date the request's clock falls on. In poller ACs, evaluated in `cfg.Location`. In Lambda ACs, evaluated in `sydneyTZ` as the existing handlers already do.
- **yesterday**: the calendar date one day before "today" in `cfg.Location`. On DST transitions the poller's standard `time.Time.AddDate(0, 0, -1)` semantics apply (the same arithmetic the existing `pollDailyEnergy` loop uses for its yesterday retry).
- **summarisation pass**: the new poller routine that, for one (date, system) pair, queries the day's readings, computes derivedStats, and writes them to `flux-daily-energy` via `UpdateItem`.
- **derivedStats package**: the shared Go package introduced by this spec that hosts `findDailyUsage`, `findMinSOC`, and `findPeakPeriods` so that both `internal/poller` and `internal/api` can import the helpers without one depending on the other.

## 1. Poller: Hourly Summarisation Pass

**User Story:** As the system maintainer, I want the poller to summarise yesterday's readings into per-day derived stats every hour, so that completed dates carry the stats on disk before their readings age out of the 30-day TTL window.

**Acceptance Criteria:**

1. <a name="1.1"></a>The poller SHALL run a summarisation pass once per hour against the date that is "yesterday" in `cfg.Location` at the moment the pass fires. The first tick of the loop SHALL fire immediately on poller startup (matching the existing `pollLoop` behaviour) so that a container restart inside the post-midnight gap closes the gap on its first tick.  
2. <a name="1.2"></a>The summarisation pass SHALL NOT run against today; today's derivedStats are produced live by `/day` and `/history` on demand.  
3. <a name="1.3"></a>The summarisation pass SHALL fetch the day's readings from `flux-readings` for the date being summarised, invoke `findDailyUsage`, `findMinSOC`, and `findPeakPeriods` against them using the same SSM-resolved off-peak window the Lambda uses, and write the three results to the `(sysSn, date)` row in `flux-daily-energy`.  
4. <a name="1.4"></a>The pass SHALL use a DynamoDB `UpdateItem` with a field-level `SET` expression that touches only the three derived attributes, leaving every other attribute on the row (the AlphaESS-sourced energy totals, any future fields) untouched. The pass SHALL be update-only â€” when the row does not yet exist for the date, the pass SHALL skip the write and let the next AlphaESS energy poll create the row.  
5. <a name="1.5"></a>WHEN no readings exist for the date being summarised, the pass SHALL skip the write entirely (no `UpdateItem` call) and log at info level. Existing attribute values on the row SHALL NOT be cleared as a side effect.  
6. <a name="1.6"></a>WHEN the off-peak SSM parameters are unresolved at pass time, the pass SHALL skip the write entirely (no `UpdateItem` call) and log at warn level with the date being skipped. The next hourly tick will retry once SSM resolves. This preserves the past-dates-immutable invariant by ensuring the row is never written with a degraded `dailyUsage` shape that a later tick would have to correct.  
7. <a name="1.7"></a>WHEN the readings query, computation, or `UpdateItem` call fails, the failure SHALL be logged at error level with the date and system serial; the pass SHALL NOT crash the poller process or affect the other ticker loops.  
8. <a name="1.8"></a>The summarisation pass for a (date, system) pair SHALL be deterministic: re-running it against the same readings and the same SSM-resolved off-peak window SHALL produce field-equivalent and value-equivalent UpdateItem payloads (same field set, same scalar values for each field). Byte-equivalent JSON / DynamoDB-AttributeValue marshalling is NOT asserted (Go map iteration order and `attributevalue` marshalling do not guarantee it).  
9. <a name="1.9"></a>The pass SHALL share a single computation entry point with the Lambda's live path. The three helpers SHALL live in a shared **derivedStats package** that both `internal/poller` and `internal/api` import; neither package SHALL depend on the other for the helpers.  
10. <a name="1.10"></a>BEFORE issuing the readings query in [1.3](#1.3), the pass SHALL `GetItem` the `(sysSn, date)` row and SHALL skip the readings query, the computation, and the `UpdateItem` call WHEN a `derivedStatsComputedAt` sentinel attribute is present on the row. The sentinel is set by the pass alongside the three derived attributes in the same `UpdateItem` SET expression, so its presence is equivalent to "this date has been summarised at least once". The sentinel approach is preferred over inspecting the three derived attributes directly because (a) `findDailyUsage` legitimately returns `nil` when no blocks survive the pipeline (the date would otherwise re-compute every hour forever), and (b) `findPeakPeriods` legitimately returns an empty slice on cloudy days with no excursions (a present-but-empty list cannot be distinguished from "never written" by attribute-shape inspection alone). A row carrying any of the three derived attributes without the sentinel (e.g. from a partially-failed prior pass that wrote two-of-three before crashing â€” extremely unlikely with the single-SET semantics of [1.4](#1.4) but kept as a defensive case) SHALL NOT short-circuit; the pass SHALL re-compute and write the full set including the sentinel.  
11. <a name="1.11"></a>The poller SHALL emit a CloudWatch custom metric named `SummarisationPassResult` with dimensions `Result âˆˆ {success, skipped-no-readings, skipped-no-row, skipped-ssm-unresolved, skipped-already-populated, error}` per pass execution, so the operator can alarm on a derivedStats row that remains absent more than 24 hours after its date completed. The poller's IAM role SHALL be granted `cloudwatch:PutMetricData` on the `Flux/Poller` namespace; this grant is part of the same atomic deploy as Decision 3's `UpdateItem` permission (per Section 7).  
12. <a name="1.12"></a>The poller SHALL operate against the single configured system serial (`cfg.Serial`) â€” the (sysSn, date) keying on disk preserves a future multi-system extension but no list iteration is added by this spec.  
13. <a name="1.13"></a>WHEN the summarisation pass fires before that day's energy row exists for yesterday (a fresh container started after midnight, with the energy poll for yesterday not yet completed), AC [1.4](#1.4)'s skip applies. The worst-case time-to-populate is bounded by the next hourly tick of both this pass and the existing `pollDailyEnergy` loop â€” at most one hour after both have succeeded for the date. This window is the operational cost of the immutability invariant from Decision 2.  
14. <a name="1.14"></a>The off-peak SSM resolution used by [1.3](#1.3) and gated by [1.6](#1.6) SHALL come from the same SSM-fetch mechanism the rest of the poller uses (currently a fresh fetch per request via the AWS SDK's default behaviour; if a cached resolver is later introduced, the same one SHALL serve this pass). The next hourly tick after an unresolved tick re-reads SSM via the same path; no per-tick blocking or retry-with-backoff is added by this spec.  

## 2. DynamoDB: Schema Extension

**User Story:** As a developer reading the table downstream, I want the new attributes on `flux-daily-energy` to be optional, native, and self-describing, so that pre-existing rows continue to deserialise without migration and the AWS console remains useful for ad-hoc inspection.

**Acceptance Criteria:**

1. <a name="2.1"></a>The `flux-daily-energy` row SHALL gain four new optional attributes â€” one carrying the `findDailyUsage` payload, one carrying the `findMinSOC` payload (`socLow` and `socLowTime`), one carrying the `findPeakPeriods` payload, and a `derivedStatsComputedAt` RFC3339 UTC timestamp string acting as the precheck sentinel per AC [1.10](#1.10) â€” stored as native DynamoDB Map / List / String attributes via `attributevalue.MarshalMap` (NOT as JSON-encoded strings). This keeps the data inspectable in the AWS console and avoids baking a serialisation choice into the storage layer.  
2. <a name="2.2"></a>Pre-existing rows that lack the new attributes SHALL deserialise into the Go model with the corresponding fields set to their zero / nil value.  
3. <a name="2.3"></a>No existing attribute on `flux-daily-energy` SHALL be removed, renamed, or have its DynamoDB type changed.  
4. <a name="2.4"></a>No DynamoDB GSI, LSI, or table-level setting (TTL, billing mode, stream config) SHALL be changed by this feature.  
5. <a name="2.5"></a>The implementation SHALL measure the post-feature `flux-daily-energy` row size on a representative day from the live table (a typical clear-sky weekday) and confirm the row stays below the 4 KB single-item read-unit boundary so that a `GetItem` for one row continues to consume one RCU. The pre-feature row is ~200 B; the budget for the three new attributes is therefore ~3.8 KB. If the measurement exceeds the budget, the design SHALL be revisited (smaller block / period payloads, or a JSON-string fallback per Decision 4's rejected alternative) rather than the AC silently relaxed.  

## 3. Lambda `/day`: Pre-computed for Past, Live for Today

**User Story:** As an iOS app user, I want `/day` for a past date to return the same shape it returns today, so that the Day Detail screen doesn't change behaviour between dates.

**Acceptance Criteria:**

1. <a name="3.1"></a>WHEN `/day` is called for a completed date, the handler SHALL read derivedStats from the `flux-daily-energy` row and serve them in the response in the same shape they previously had when computed live (`dailyUsage` object, `summary.socLow` / `summary.socLowTime`, `peakPeriods` array).  
2. <a name="3.2"></a>WHEN `/day` is called for today, the handler SHALL compute derivedStats live from the day's readings exactly as it does today; no behavioural change SHALL occur for the today path.  
3. <a name="3.3"></a>WHEN `/day` is called for a completed date whose row is missing one or more of the three derived attributes (pre-feature row, summarisation pass failed, or SSM unresolved at every prior tick), the response SHALL omit only the missing sections (the `dailyUsage` object, the `socLow`/`socLowTime` fields, the `peakPeriods` array) following the same omission rules `/day` already uses for those sections; the rest of the response SHALL serve normally.  
4. <a name="3.4"></a>The handler SHALL NOT attempt to fall back to live computation for completed dates whose derivedStats are absent (readings may have aged out, and a partial-readings live recomputation would yield wrong results).  
5. <a name="3.5"></a>The handler SHALL NOT issue a `flux-readings` `QueryReadings` call for completed dates (the readings query is now exclusive to the today path). The existing `flux-daily-power` fallback path (`QueryDailyPower` + `mapDailyPowerToPoints` + `findMinSOCFromPower`) SHALL be preserved unchanged for completed dates whose readings have aged out â€” the time-series chart and the daily-power-derived `socLow` continue to work for old dates as they do today.  
6. <a name="3.6"></a>The serialised `dailyUsage` payload returned for a completed date SHALL be field-equivalent to the one the live path would have produced from the same readings (same set of blocks, same `kind`/`start`/`end`/`totalKwh`/`averageKwhPerHour`/`percentOfDay`/`status`/`boundarySource` values). Field-equivalence (not byte-equivalence) is the contract â€” float serialisation and JSON ordering are allowed to differ.  
7. <a name="3.7"></a>The clock-of-record for the completed-vs-today bucketing decision SHALL be the value of `h.nowFunc()` evaluated once at request entry. A `/day` call whose clock falls within seconds of midnight Sydney SHALL bucket consistently within that single request (no second read of the clock for the same date).  

## 4. Lambda `/history`: Surface Derived Stats per Day

**User Story:** As an iOS client, I want each day in the history range to carry its own derivedStats, so that multi-day rollup features can render without per-day API calls.

**Acceptance Criteria:**

1. <a name="4.1"></a>The `/history` response SHALL include, on each `DayEnergy` row in the requested range, the same three derivedStats sections as `/day`: `dailyUsage`, `socLow`/`socLowTime`, `peakPeriods`.  
2. <a name="4.2"></a>For completed dates within the range, the values SHALL be read from `flux-daily-energy` (no readings query for those dates).  
3. <a name="4.3"></a>For today's row in the range, the handler SHALL compute derivedStats live from the day's readings the same way `/day` does for today, so that the today bar is up-to-date and consistent with the dashboard. The today readings query is the same one the handler already issues for energy reconciliation (see `handleHistory` in `internal/api/history.go` â€” the readings slice is held in `allReadings` for the lifetime of the handler call); the marginal cost is three additional in-memory passes (`findDailyUsage`, `findMinSOC`, `findPeakPeriods`) over that slice.  
4. <a name="4.4"></a>For completed dates whose row is missing one or more derived attributes, the corresponding sections on that day's `DayEnergy` row SHALL be omitted; other days in the range SHALL be unaffected.  
5. <a name="4.5"></a>No existing `DayEnergy` field SHALL be removed, renamed, or have its type changed.  
6. <a name="4.6"></a>The clock-of-record for the completed-vs-today bucketing decision SHALL be the value of `h.nowFunc()` evaluated once at request entry, the same instant used to compute `today` and `startDate`. A `/history` call SHALL bucket every date in its range against that single instant.  
7. <a name="4.7"></a>`/history` p95 latency SHALL be measured pre- and post-feature on a 30-day request against a warm Lambda. The post-feature p95 SHALL stay within +50 ms of the pre-feature baseline; if the measurement exceeds the budget, the AC SHALL be re-evaluated rather than the implementation rushed to fit a guess.  
8. <a name="4.8"></a>The serialised payload size of a 30-day `/history` response SHALL be measured pre- and post-feature. The expected post-feature size is ~30 KB â†’ ~60 KB on a typical workload (dominated by the three new sections per row); a measured size more than 3Ã— the pre-feature baseline SHALL trigger a design review of the attribute shape rather than acceptance of the larger payload.  
9. <a name="4.9"></a>WHEN the today-row readings query in [4.3](#4.3) fails, the handler SHALL log at warn level, omit the today row's `dailyUsage`, `socLow`/`socLowTime`, and `peakPeriods` sections from the response, and serve the rest of the range normally. The today row's existing energy totals (sourced from the daily-energy query, not the readings query) SHALL still be returned. A failed today readings query SHALL NOT fail the entire `/history` request.  
10. <a name="4.10"></a>For the cross-handler equivalence contract in [3.6](#3.6) and [6.6](#6.6), an off-peak SSM configuration change in flight (the Lambda's SSM cache holding the new value while the poller's cache still holds the old, or vice versa) MAY produce derivedStats values for the same date that differ between the two handlers. This is a known gap in the equivalence contract and is acceptable because off-peak SSM changes are operator-driven and rare; the next summarisation pass after the poller picks up the new value re-establishes equivalence.  

## 5. iOS: Model Decoding

**User Story:** As the iOS client, I want to decode the new fields when present and fall through cleanly when absent, so that adding the History Daily Usage card later is a UI-only change.

**Acceptance Criteria:**

1. <a name="5.1"></a>The Swift `DayEnergy` struct in `FluxCore` SHALL gain optional properties for the three derivedStats sections, mirroring the JSON shape of the `/day` response (the existing `DailyUsage` and `DailyUsageBlock` types are reused; new `Codable` types SHALL be introduced for `socLow`/`socLowTime` and `peakPeriods` only if equivalents do not already exist).  
2. <a name="5.2"></a>Existing `DayEnergy` decoding SHALL continue to succeed against historic JSON payloads that lack the new fields (the new properties decode to `nil`).  
3. <a name="5.3"></a>No existing `DayEnergy` property SHALL be removed, renamed, or have its type changed.  
4. <a name="5.4"></a>The on-device cache (`CachedDayEnergy`) SHALL persist the three new sections by adding three new optional `Codable`-conforming properties to the `@Model` and round-tripping them through `init(from:)` and `asDayEnergy` without loss. The persisted shape SHALL NOT introduce any new `@Model` relationship or child entity (a single `@Model` with optional Codable properties is the chosen mechanism to keep the SwiftData migration story trivial).  
5. <a name="5.5"></a>Pre-existing cached rows SHALL load with the new sections set to `nil`. The implementation SHALL verify this empirically against an existing cache file (loaded on a simulator with pre-feature state) rather than asserting it from documentation, since SwiftData migration behaviour for new optional properties is a "usually no-op" rather than a contract. IF the empirical verification fails (i.e. SwiftData rejects the new schema as incompatible), the implementation SHALL add a SwiftData lightweight migration mapping the pre-feature `CachedDayEnergy` schema version to the post-feature one with the new properties defaulting to `nil`. This migration path SHALL be in scope for this spec; deferring it would leave existing users with a broken cache on first launch of the new build.  
6. <a name="5.6"></a>This spec SHALL NOT add any new SwiftUI view, view-model field, or chart â€” UI consumption of the new fields is the responsibility of T-1022.  

## 6. Testing

**User Story:** As the project maintainer, I want enough coverage on the new write path and the read-branch logic to trust the migration, so that the spec lands without silent data drift between live and stored payloads.

**Acceptance Criteria:**

1. <a name="6.1"></a>Poller unit tests SHALL cover: a successful summarisation pass that writes all three fields; a pass against a date with no readings (no `UpdateItem` call, no row mutation); a pass against a date with no pre-existing energy row (skipped per [1.4](#1.4)); a pass against a date with off-peak SSM unresolved (skipped per [1.6](#1.6)); a pass where the readings query fails (logged, no panic, no write); a pass where the `UpdateItem` fails (logged, no panic); a pass where the row already carries all three derived attributes (skipped per [1.10](#1.10) with no readings query issued); a pass invoking the helpers with the date being summarised passed as `today` to `findDailyUsage` so the function takes the completed-day branch (no today-gate firing).  
2. <a name="6.2"></a>Poller unit tests SHALL assert that two consecutive passes against the same readings and same SSM window produce field-equivalent and value-equivalent `UpdateItem` payloads (per [1.8](#1.8)) and that the second pass is short-circuited by the [1.10](#1.10) precheck.  
3. <a name="6.3"></a>DynamoDB unit tests SHALL assert that a row written by the new `UpdateItem`-based energy writer (per Decision 3) deserialises with the three new fields nil; that a row subsequently updated by the summarisation `UpdateItem` path deserialises with both sets of fields present; and that a pre-feature legacy `PutItem`-written row (a fixture representing rows that exist on disk before the deploy) deserialises identically when read by the post-feature code. The legacy `PutItem` writer itself is retired by Decision 3 / Section 7's atomic deploy and SHALL NOT be exercised by post-feature tests beyond the pre-feature-fixture read.  
4. <a name="6.4"></a>Lambda `/day` tests SHALL cover: completed-date with all three fields present (served from storage); completed-date with one field absent (only that section omitted); completed-date with all three absent (all sections omitted); completed-date with no readings AND no derivedStats but `flux-daily-power` available (the time-series chart and `socLow` from daily-power continue to render â€” regression test for AC [3.5](#3.5)); today (live compute, unchanged from pre-feature behaviour); a regression test asserting `/day` for a completed date issues no `QueryReadings` call.  
5. <a name="6.5"></a>Lambda `/history` tests SHALL cover: a 7-day window where every day has all three fields; a window where the oldest day lacks derived fields; a window straddling today (today's row carries live-computed derivedStats, past rows carry stored).  
6. <a name="6.6"></a>A cross-handler equivalence test SHALL assert that, for a completed date whose row carries derivedStats, `/day` and `/history` return field-equivalent derivedStats payloads (same field set, same values), evaluated against a single fixed clock so the bucketing decision is identical between the two handlers.  
7. <a name="6.7"></a>An end-to-end test SHALL exercise the full pollerâ†’DynamoDBâ†’Lambda round-trip: stage a day's worth of readings in a real DynamoDB surface (DynamoDB Local in CI, or a Testcontainers-managed image â€” NOT a hand-rolled in-memory fake), invoke the summarisation pass, then invoke `/day` and `/history` against the same date and assert the responses carry the expected derivedStats shape. The real-DynamoDB requirement is load-bearing: the failure class this AC catches is attribute-shape mismatches between writer and reader, and a hand-rolled fake that re-implements `UpdateItem` with `SET` against Map / List attributes will not faithfully reproduce DynamoDB's marshalling behaviour.  
8. <a name="6.8"></a>iOS `FluxCore` decoding tests SHALL cover: a `DayEnergy` JSON with all three new sections (decodes with non-nil properties); a `DayEnergy` JSON with none of them (decodes with nil properties, no error); the `CachedDayEnergy` round-trip for both cases; loading a cache file written by the pre-feature build (verifies the no-migration claim from [5.5](#5.5) empirically).  

## 7. Deployment

**User Story:** As the operator deploying the change, I want the writer migration to roll out atomically, so that an in-flight deploy cannot have an old PutItem-based writer clobber new derived fields written by the summarisation pass.

**Acceptance Criteria:**

1. <a name="7.1"></a>The poller binary deployed to ECS SHALL contain both the new energy writer (UpdateItem-based, per Decision 3) and the new summarisation pass in the same image. The two writers SHALL NOT be deployed in separate releases; a single ECS task definition update SHALL switch both at once. The current deployment runs `desiredCount=1`, so an atomic swap is feasible without coordination.  
2. <a name="7.2"></a>The Lambda deployment carrying the storage-read changes (`/day` and `/history` reading derivedStats from storage) MAY be deployed independently of the poller image â€” the read path tolerates missing derived attributes per AC [3.3](#3.3) and [4.4](#4.4), so deploying the Lambda before the first successful summarisation pass is safe.  
3. <a name="7.3"></a>The CloudFormation template SHALL grant the poller's ECS task role two new IAM permissions on the same atomic deploy as the binary swap in [7.1](#7.1): `dynamodb:UpdateItem` on the `flux-daily-energy` table (per Decision 3) and `cloudwatch:PutMetricData` on the `Flux/Poller` namespace (per AC [1.11](#1.11)). Deploying the binary without the IAM grants SHALL be treated as a deploy bug; the CloudFormation update SHALL precede or land with the ECS service update.
